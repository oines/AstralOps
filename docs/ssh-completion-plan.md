# AstralOps SSH Completion Plan

Last updated: 2026-05-24

## 1. Current State Audit

AstralOps already has the right scaffolding:

- `Workspace.Target = "ssh"` and `Workspace.SSH` exist in `daemon/models.go` and `protocol/src/index.ts`.
- The desktop "new workspace" flow already collects SSH endpoint, port, and remote cwd.
- `~/.AstralOps/projections/<workspace_id>/root` is created for every workspace.
- `astralopsd` has workspace/session/event APIs and local file/exec/PTY endpoints.
- `proxy-agent` already supports JSON-line RPC for `hello`, `stat`, `list`, `read`, `write`, `mkdir`, `remove`, `glob`, `grep`, `exec`, `git_status`, and `git_diff`.

The missing pieces are the actual SSH control plane:

- `/v1/workspaces/:id/connect` currently only emits `workspace.connected`; it does not open SSH, upload/start proxy-agent, or validate remote cwd.
- `/v1/workspaces/:id/files`, `/exec`, and `/pty` reject SSH workspaces.
- Claude and Codex runtimes reject SSH workspaces before starting a turn.
- `proxy-agent` `exec` is buffered, not streaming.
- `proxy-agent` `apply_patch` and `pty_*` are intentionally unimplemented.
- No sparse projection manager exists yet.

## 2. Target Behavior

For a remote workspace, the user experience must be:

1. Create workspace with `endpoint`, `port`, `remote_cwd`, and agent.
2. AstralOps validates SSH access using the user's existing `ssh` config/key.
3. AstralOps uploads or reuses a matching `astral-proxy-agent` binary on the remote host.
4. AstralOps starts `astral-proxy-agent` over SSH stdio; no remote port is opened.
5. File browser, terminal, command execution, Claude tools, and later Codex runtime all operate against `remote_cwd`.
6. Only files touched by tools are hydrated into `local_projection_root`.
7. UI always displays remote paths, not projection paths.

## 3. Implementation Plan

### Phase A - SSH Manager

Add daemon files:

- `daemon/ssh_manager.go`
- `daemon/ssh_proxy_client.go`
- `daemon/projection.go`

Introduce an app-owned manager:

```go
type sshManager struct {
  mu sync.Mutex
  targets map[string]*sshTarget
}

type sshTarget struct {
  workspace Workspace
  proxy *proxyClient
  remoteInfo remoteHello
  state string // disconnected, connecting, connected, degraded
}
```

Use the system `ssh`/`scp` binaries instead of Go `crypto/ssh` so user `~/.ssh/config`, ProxyJump, IdentityFile, agent forwarding, ControlMaster, known_hosts, and custom aliases work naturally.

Connection command defaults:

```text
ssh -p <port> -o BatchMode=yes -o ServerAliveInterval=15 -o ServerAliveCountMax=3 <endpoint> <remote command>
scp -P <port> <local proxy-agent> <endpoint>:<remote path>
```

Do not parse or read private keys in AstralOps.

### Phase B - Remote Bootstrap

`POST /v1/workspaces/:id/connect` must:

1. Validate `workspace.target == "ssh"`.
2. Run remote probe:

```sh
printf '%s\n' "$(uname -s)" "$(uname -m)" "$SHELL" "$PWD"
test -d "$REMOTE_CWD"
test -r "$REMOTE_CWD"
test -x "$REMOTE_CWD"
```

3. Normalize OS/arch:

```text
Darwin arm64  -> darwin-arm64
Darwin x86_64 -> darwin-amd64
Linux aarch64 -> linux-arm64
Linux x86_64  -> linux-amd64
```

4. Pick local binary:

```text
bin/proxy-agent/<os>-<arch>/astral-proxy-agent
```

5. Create remote runtime dir:

```text
/tmp/.astralops/<workspace_id>/
```

6. Upload binary if checksum differs:

```sh
sha256sum /tmp/.astralops/<workspace_id>/astral-proxy-agent
```

Fallback for macOS remote:

```sh
shasum -a 256 <file>
```

7. Start proxy-agent as an SSH stdio child:

```sh
cd "$REMOTE_CWD" && /tmp/.astralops/<workspace_id>/astral-proxy-agent --cwd "$REMOTE_CWD"
```

8. Send `hello` RPC and emit:

```text
workspace.connected
workspace.proxy.ready
```

On failure emit:

```text
workspace.connection_failed
workspace.degraded
```

### Phase C - Proxy Client

Implement a long-lived JSON-line client:

```go
type proxyClient struct {
  cmd *exec.Cmd
  stdin io.WriteCloser
  stdout io.Reader
  pending map[string]chan proxyResponse
}
```

Required behavior:

- Request ids are generated by daemon.
- One goroutine scans stdout and resolves pending calls.
- stderr is converted into `workspace.proxy.warning`.
- If the SSH process exits, mark workspace `degraded`.
- A new call on a degraded target attempts reconnect once before failing.

### Phase D - Make Workspace APIs Remote-Aware

Update current local-only handlers:

- `handleWorkspaceFiles`
- `handleWorkspaceExec`
- `handleWorkspacePTY`

Behavior:

- Local workspace continues using current local implementation.
- SSH workspace routes to `sshManager.target(workspace).proxy`.

Remote file browser:

```text
GET /v1/workspaces/:id/files?path=<remote-relative-path>
```

Maps relative path under `remote_cwd`, calls proxy `list`, and returns the existing `FileListResponse` shape.

Remote command:

```text
POST /v1/workspaces/:id/exec
```

Calls proxy `exec` with cwd = `remote_cwd`.

Remote terminal:

```text
WS /v1/workspaces/:id/pty
```

Uses proxy `pty_start`, `pty_write`, `pty_resize`, `pty_kill`.

### Phase E - Proxy-Agent Completion

Update `proxy-agent/main.go`.

Add command-line flags:

```text
--cwd <path>
--max-read-bytes <n>
```

Make all file paths resolve under cwd unless absolute paths are explicitly allowed by the daemon request.

Implement missing RPCs:

```text
apply_patch
pty_start
pty_write
pty_resize
pty_kill
exec_stream_start
exec_stream_kill
```

Keep the existing buffered `exec` for simple API calls, but add streaming exec for tools/terminal output:

```json
{"id":"1","method":"exec_stream_start","params":{"cwd":"/app","command":"npm test"}}
{"id":"1","event":"stdout","data":"..."}
{"id":"1","event":"stderr","data":"..."}
{"id":"1","event":"exit","exit_code":0}
```

For PTY, use `github.com/creack/pty` in `proxy-agent` too.

For `apply_patch`, do not shell out to `patch` blindly. Implement one of:

- Preferred: reuse a small internal unified-diff parser for add/delete/update hunks.
- Acceptable V1 fallback: write patch to a temp file and run `git apply --whitespace=nowarn`, only if `git` exists and cwd is inside a git repo.

### Phase F - Sparse Projection Manager

Projection manager responsibilities:

```go
type projectionManager struct {
  workspace Workspace
  proxy *proxyClient
}
```

Operations:

- `HydrateFile(remotePath)` -> read remote, write local projection.
- `HydrateSearchResults(matches)` -> create parent dirs and hydrate matched files.
- `PushFile(localPath)` -> write local projection content back to remote path.
- `ApplyPatchToRemote(patch)` -> use proxy `apply_patch` or per-file writes.
- `RemoteToLocal(remotePath)` and `LocalToRemote(localPath)`.

Path rules:

- All display paths are remote-relative.
- Projection root is never exposed to model-visible text or UI labels except debug panels.
- Path escape attempts fail before reaching proxy-agent.

### Phase G - Claude Remote Runtime

Modify `claudeLocalRuntime.StartTurn`:

- If local: keep current behavior.
- If ssh: use projection root as `cmd.Dir`, but inject remote identity into prompt and hooks.

Required additions:

- `daemon/claude_remote_hooks.go`
- generated per-session hook config under:

```text
~/.AstralOps/workspaces/<workspace_id>/claude-hooks/
```

Start Claude with a process-local temporary settings file, not global settings.

Remote prompt injection:

```text
You are working on SSH workspace <endpoint>:<remote_cwd>.
Paths shown to the user and in final answers must be remote paths.
The local projection directory is an implementation detail and must not be mentioned.
```

Hook behavior:

- `Read`: hydrate remote file before Claude reads local projection.
- `LS`: remote list, mirror minimal directory entries.
- `Glob`: remote glob, create sparse local paths.
- `Grep`: remote grep, hydrate matching files.
- `Bash`: execute via proxy-agent in remote cwd; return command result to Claude.
- `Write/Edit/MultiEdit`: let Claude mutate projection, then compute local diff and request approval before remote push.

V1 simplification:

- Do not try to unblock the same Claude interactive turn for Ask/Plan until SDK control protocol is implemented.
- Continue current follow-up turn strategy for Claude Ask/Plan/permission.

### Phase H - Codex Remote Strategy

Do not block SSH completion on Codex remote adapter.

For V1 remote SSH completion:

- Claude remote must be fully functional first.
- Codex remote workspace creation may be allowed but should show `control.error` explaining that Codex remote runtime is not ready.

After Claude remote is stable:

- Add `codex_remote_runtime.go`.
- Keep using user stock Codex config/auth.
- Either fork/patch Codex runtime backend or run a controlled app-server shim that translates file/shell operations through `proxyClient`.

Acceptance for Codex remote is separate from SSH workspace completion.

## 4. API Changes

Extend health/workspace connection state:

```ts
type WorkspaceConnection = {
  workspace_id: string
  status: "disconnected" | "connecting" | "connected" | "degraded" | "failed"
  target: "local" | "ssh"
  remote?: {
    endpoint: string
    cwd: string
    os: string
    arch: string
    proxy_version: string
  }
  message?: string
}
```

New endpoints:

```text
GET  /v1/workspaces/:id/connection
POST /v1/workspaces/:id/disconnect
POST /v1/workspaces/:id/projection/hydrate
POST /v1/workspaces/:id/projection/push
GET  /v1/workspaces/:id/projection/files
```

Existing endpoints become remote-aware:

```text
GET  /v1/workspaces/:id/files
POST /v1/workspaces/:id/exec
WS   /v1/workspaces/:id/pty
```

New events:

```text
workspace.connecting
workspace.connected
workspace.connection_failed
workspace.degraded
workspace.reconnected
workspace.proxy.ready
workspace.proxy.warning
workspace.proxy.exited
workspace.projection.hydrated
workspace.projection.pushed
tool.output_delta
tool.exit
```

## 5. UI Changes

New workspace modal:

- After SSH endpoint input, call `connect` before allowing session creation.
- Show probe result: OS/arch/shell/remote cwd.
- Remote directory selection can be text input first; browser can use remote `files` endpoint once connected.

Right panel:

- File tab works for SSH by calling remote-aware `/files`.
- Terminal tab works for SSH by calling remote-aware `/pty`.
- Add connection badge states: connecting, connected, degraded, failed.

Transcript/tool timeline:

- Command output should stream for remote commands.
- File changes show remote paths.
- Projection drawer shows hydrated files and dirty files.

## 6. Tests

Unit tests:

- SSH command construction preserves endpoint aliases and port.
- OS/arch normalization.
- Proxy JSON-line request/response routing.
- Proxy reconnect on exited process.
- Remote path mapping cannot escape remote cwd.
- Projection hydrate/push path mapping.
- Proxy-agent `apply_patch`.
- Proxy-agent PTY lifecycle.

Integration tests without a real remote host:

- Start `proxy-agent` as a local subprocess and treat it as remote.
- Create SSH workspace with a fake `ssh` command script that launches local proxy-agent.
- Verify `/connect`, `/files`, `/exec`, `/pty`.
- Verify degraded event when proxy exits.

Manual acceptance:

1. Create SSH workspace `root@host:/tmp/project`.
2. File browser lists remote files.
3. Terminal opens remote shell and `pwd` prints remote cwd.
4. `POST /exec` runs remotely.
5. Claude remote session can read a remote file.
6. Claude remote session can grep remote content.
7. Claude remote session can edit a remote file after approval.
8. Disconnect network/kill proxy-agent; UI shows degraded.
9. Reconnect; remote workspace recovers without losing local UI event history.

## 7. Suggested Build Order

1. Implement `sshManager` and remote `connect`.
2. Add proxy-client JSON-line transport and degraded/reconnect events.
3. Make `/files` and `/exec` remote-aware.
4. Implement proxy-agent streaming exec.
5. Implement remote PTY.
6. Implement sparse projection manager.
7. Implement Claude remote Read/LS/Glob/Grep/Bash projection hooks.
8. Implement Claude Write/Edit/MultiEdit approval and push.
9. Add UI connection/projection states.
10. Only then start Codex remote adapter.

## 8. Non-Goals For This SSH Pass

- Mobile pairing.
- Multi-desktop routing.
- End-to-end encryption.
- IM bot integration.
- Remote installation of Claude/Codex.
- Full Codex remote runtime parity.
- Container sandbox isolation.
